# User guide

For the dev who just cloned the template and wants to ship something. Skip what you already know.

## First run

```bash
git clone <this-repo> my-app
cd my-app
npm install
cp .env.example .env       # leave empty for first run — everything no-ops gracefully
npm run db:generate        # produces drizzle/0000_*.sql
npx expo start
```

Press `i` for iOS Simulator, `a` for Android emulator, `w` for web.

The app boots with all third-party flags `false`. You can use the home tab (local data) and the language toggle. Auth, paywall, sync, analytics, crash — all off until you fill `.env`.

## Two paths from here

### A) Use this directly to build a specific app yourself

You skip `/generate-app`, you write features by hand using the template and the rules.

→ Read [`docs/architecture.md`](architecture.md), [`docs/data-sync.md`](data-sync.md), [`docs/design-system.md`](design-system.md), [`docs/components/README.md`](components/README.md).

→ Then jump to "Adding a feature" below.

### B) Run the Claude pipeline to scaffold a tailored app

You describe an idea, the pipeline generates branding, design, schema, screens, copy, backend setup.

```
/generate-app
```

Answer the prompts. The pipeline pauses 4 times for GO / REFAIRE / AJUSTER. At the end you get a tailored app + filled `.planning/` folder + a setup checklist.

→ Read [`README.md`](../README.md) "Generating a new app" section.

## Adding a feature — the recipe

Say you want to add a "favorite items" feature.

### 1. Decide free or premium

Default rule: capture is free, scale + sync + exports are premium.

"Mark as favorite" — a capture-style action → free.
"Sort by favorites in cloud history" — depends on cloud → premium.

### 2. Schema

If it's data, model it. Edit `lib/db/schema.ts`:

```ts
export const items = sqliteTable("items", {
  // ... existing columns ...
  isFavorite: integer("is_favorite", { mode: "boolean" }).notNull().default(false),
});
```

Generate the migration:

```bash
npm run db:generate
```

Commit the new file in `drizzle/`.

If you added a NEW table (not just a column), see [`docs/data-sync.md`](data-sync.md) "Adding a new synced table" for the full Postgres + RLS work.

### 3. UI

Reuse primitives. New screen → file in `app/(tabs)/` (or wherever it routes). New component → `components/{domain}/`.

For a premium feature, wrap with the gate:

```tsx
import { PremiumGate } from "@/components/paywall/PremiumGate";

<PremiumGate feature="favorites_in_history">
  <FavoritesView />
</PremiumGate>
```

For data writes that should be free locally but synced only when premium: do nothing special — the sync engine guards itself.

### 4. Copy

Add keys to BOTH `messages/fr.json` AND `messages/en.json`. The hook will block if you forget one.

```json
{ "items": { "favorite": "Favori", "unfavorite": "Retirer" } }
```

### 5. Analytics (only if it's a tracked outcome)

If "favorited" is a meaningful outcome (not just a click), add to the allowlist:

```ts
// lib/analytics/events.ts
export const ANALYTICS_EVENTS = {
  // ...
  feature_used: "feature_used",  // already exists, use it
};
```

Then track:

```ts
import { track } from "@/lib/analytics/posthog";
track("feature_used", { feature: "favorited", context: "home_list" });
```

Don't add a new top-level event for a sub-feature — `feature_used` with `feature: "..."` is the catch-all.

### 6. Type-check

```bash
npm run typecheck
```

The PostToolUse hook also runs this on every Edit/Write so you'll see errors as soon as they appear.

### 7. Test

```bash
npx expo start
```

Tap through the new feature in both light and dark mode. Toggle locale.

## When to set up the cloud

The first time a user actually needs an account:
- Premium feature wants sync.
- You want analytics.
- You want crash reports.

Follow [`.planning/backend-setup.md`](../.planning/) (generated by the `/generate-app` pipeline) OR the manual checklist in `.claude/agents/backend-provisioner.md`.

In short:
1. **Supabase**: free project, paste the DDL + RLS from `.planning/db-schema.md`.
2. **RevenueCat**: free project, create entitlement `premium`, two products, default offering.
3. **Sentry**: free project, copy DSN.
4. **PostHog**: free EU project, copy key.
5. **EAS**: `npx eas-cli login && npx eas init --id`.

Add the keys to `.env`. Restart the dev server. Each `featureFlags.X` becomes `true` automatically.

## Building for stores

```bash
# dev client (one-time per native change)
npx expo prebuild --clean
ALLOW_PAID=0 npx eas-cli build --profile development --platform ios

# preview build (TestFlight / Play internal)
npx eas-cli build --profile preview --platform ios

# production build (the billing-guardrail blocks this without ALLOW_PAID=1)
ALLOW_PAID=1 npx eas-cli build --profile production --platform ios
```

For the full ship checklist, see `.claude/skills/expo-eas-deploy/SKILL.md` § "Production checklist before submit".

## Common workflows

### Change the palette

Edit `global.css` only. Both `:root` and `.dark` blocks. See [`docs/design-system.md`](design-system.md).

### Add a tab

```tsx
// app/(tabs)/_layout.tsx
<Tabs.Screen name="library" options={{ title: t("tabs.library") }} />
```

Create `app/(tabs)/library.tsx`. Add `tabs.library` to both messages files.

### Sign-in flow

Already wired. `app/auth/sign-in.tsx`. Routes:
- Settings → Sign in (when no session) → `auth/sign-in`
- New user → `auth/sign-up`
- Magic link → email link with `apptemplate://` scheme

For OAuth (Google, Apple), Supabase handles it but you need to:
1. Configure providers in Supabase dashboard → Authentication → Providers.
2. Add deep link handling in `app/_layout.tsx` for the OAuth callback.

Not in v1 of this template — when you need it, follow Supabase's RN OAuth guide.

### Run a sync manually

Settings → "Sync now" button. Or programmatically:

```ts
import { syncItems } from "@/lib/db/sync";
const { pushed, pulled } = await syncItems(user.id);
```

The function is a no-op when not premium or not connected. See [`docs/data-sync.md`](data-sync.md).

### Test the paywall in sandbox

iOS:
1. App Store Connect → Users → Sandbox tester. Create one.
2. On simulator/device: Settings app → App Store → Sandbox Account → sign in as tester.
3. Build a dev client, tap "Upgrade" in the app.
4. Subscriptions in sandbox auto-renew every few minutes — useful for testing.

Android:
1. Play Console → Internal testing track → add tester emails.
2. Install via the internal-testing link.
3. Test purchases use Google's test instruments.

### Audit costs

```
/cost-audit
```

Hits each service's API where credentials are configured, writes a report to `.planning/cost-audit-{date}.md`. Run before each production push.

## Troubleshooting

### "Unable to resolve module"

```bash
npx expo start -c          # clears Metro cache
```

### Migrations didn't run

Confirm `await ensureMigrations()` is awaited in `app/_layout.tsx` BEFORE the first query. Already wired — only relevant if you reorganized init.

### NativeWind classes not applied

See [`docs/design-system.md`](design-system.md) § "Debugging".

### `npm run typecheck` fails on a fresh clone

Sometimes the migration stub `drizzle/migrations.ts` has a TS issue if you haven't run `npm run db:generate` yet. Run it.

### `useLiveQuery` returns empty

`enableChangeListener: true` must be on `openDatabaseSync`. Already set in `lib/db/client.ts` — only relevant if you forked the file.

## Where things live (cheat sheet)

| I want to... | Edit... |
|---|---|
| Change the palette | `global.css` |
| Add a token | `global.css` + `tailwind.config.js` |
| Add a screen | `app/...` (mirrors expo-router) |
| Add a component | `components/ui/` (primitive) or `components/{domain}/` |
| Add a table | `lib/db/schema.ts` + `npm run db:generate` |
| Add an analytics event | `lib/analytics/events.ts` |
| Add an i18n key | `messages/{fr,en}.json` (both!) |
| Change app name / bundle id | `app.json` |
| Add an Expo plugin | `app.json` plugins + `npx expo prebuild --clean` |
| Add a permission for Claude (Bash command, etc.) | `.claude/settings.local.json` |
