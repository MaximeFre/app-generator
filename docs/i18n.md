# i18n

How to add, edit, and use translations.

## The model

- Two locale files: `messages/fr.json` and `messages/en.json`.
- Dot-notation keys: `auth.signIn`, `paywall.title`, `home.empty`.
- One function in code: `t("auth.signIn")`.
- A **PostToolUse hook** (`.claude/scripts/check-i18n-parity.sh`) blocks any write that breaks parity between the two files.

The template defaults to FR auto-detected from `expo-localization`, falls back to EN.

## Reading

In any component or screen:

```tsx
import { t } from "@/lib/i18n";

<Text>{t("home.title")}</Text>
<Text>{t("settings.syncResult", { pulled: 3, pushed: 5 })}</Text>
```

Interpolation: `{name}` placeholders in the JSON value, replaced by the params object.

```json
{ "settings": { "syncResult": "Synced {pulled} from cloud, {pushed} to cloud." } }
```

## Adding a key

1. Add to BOTH `messages/fr.json` AND `messages/en.json`. Same key path. Different values.

   ```json
   // messages/fr.json
   { "home": { "shareItem": "Partager" } }

   // messages/en.json
   { "home": { "shareItem": "Share" } }
   ```

2. Save both. The PostToolUse hook will fire after the second write — if the key exists in only one file, it blocks with:

   ```
   i18n parity broken between messages/fr.json and messages/en.json:
   Missing in en.json: home.shareItem
   ```

3. Use in code: `t("home.shareItem")`.

## Removing a key

Remove from BOTH files. Same hook applies.

## Renaming

1. Rename in both files.
2. Update every `t("old.key")` → `t("new.key")` in code. The hook doesn't catch dangling references (TypeScript can, with a small extension — see "Type-safe keys" below).

## Locale switch

In settings (already wired):

```tsx
import { useI18n } from "@/lib/i18n";

const { locale, setLocale } = useI18n();

<Button label="FR" onPress={() => setLocale("fr")} />
<Button label="EN" onPress={() => setLocale("en")} />
```

Changing locale re-renders any component using `t()` (because `t()` reads from the Zustand store).

## Auto-detection

On first launch, `lib/i18n/index.ts` reads `expo-localization`'s `getLocales()[0].languageCode`:

```ts
const detectLocale = (): Locale => {
  const code = getLocales()[0]?.languageCode?.toLowerCase();
  return code === "fr" ? "fr" : "en";
};
```

Anything other than `fr` falls back to `en`. To support a third locale (e.g., `es`), extend the type and the `dictionaries` map:

1. Create `messages/es.json` with the same keys.
2. In `lib/i18n/index.ts`:
   ```ts
   type Locale = "fr" | "en" | "es";
   const dictionaries: Record<Locale, Messages> = { fr, en, es };
   const detectLocale = (): Locale => {
     const code = getLocales()[0]?.languageCode?.toLowerCase();
     if (code === "fr") return "fr";
     if (code === "es") return "es";
     return "en";
   };
   ```
3. Update the parity hook to check all three files (currently only fr ↔ en).

## Adding a third file to the parity check

Edit `.claude/scripts/check-i18n-parity.sh`. The Python comparison currently does fr vs en — generalize to compare every `messages/*.json` against the union of all keys.

## Voice rules

The actual content rules live in `.claude/rules/content.md`. Summary:

- Direct, friendly, present tense.
- No em-dash (`—`). Use comma or period.
- No exclamation marks except confirmation toasts.
- ≤ 8 words for paywall hero, ≤ 14 for subtitle, ≤ 3 for CTAs.
- FR and EN are not literal translations — adapt idioms.

## Type-safe keys (optional)

`t()` accepts any `string` today. To get autocompletion + compile-time safety:

```ts
// lib/i18n/types.ts
import en from "@/messages/en.json";
type Paths<T, P extends string = ""> =
  T extends object
    ? { [K in keyof T & string]: Paths<T[K], P extends "" ? K : `${P}.${K}`> }[keyof T & string]
    : P;
export type I18nKey = Paths<typeof en>;

// lib/i18n/index.ts
export function t(key: I18nKey, params?: Record<string, string | number>): string { ... }
```

After this, `t("home.foo")` autocompletes; `t("typo")` is a TS error.

Skip this in v1 unless your key set grows past ~50 entries. Then it pays off.

## Storing locale across sessions

Currently the locale resets to auto-detected on every cold start. To persist:

```ts
// lib/i18n/index.ts
import { create } from "zustand";
import { persist, createJSONStorage } from "zustand/middleware";
import AsyncStorage from "@react-native-async-storage/async-storage";

export const useI18n = create<I18nState>()(
  persist(
    (set) => ({ locale: detectLocale(), setLocale: (locale) => set({ locale }) }),
    { name: "i18n", storage: createJSONStorage(() => AsyncStorage) },
  ),
);
```

Trade-off: persisting means a user who switched their device locale won't see the app's locale follow. Up to you.

## RTL support (Arabic, Hebrew)

Not handled by default. To add:

1. Detect RTL: `import { I18nManager } from "react-native"; I18nManager.isRTL`.
2. Force RTL on the dictionary load: `I18nManager.forceRTL(detectLocale() === "ar")`.
3. Restart the app (RTL forces a JS bundle restart on iOS).
4. Audit every screen — flex direction, padding, icons need to mirror.

Don't add RTL until you have a real RTL user — it's a multi-day audit, not a one-line config.

## Plurals

Not built-in. For `0 / 1 / many`:

```ts
function pluralize(count: number, key: string) {
  if (count === 0) return t(`${key}.zero`);
  if (count === 1) return t(`${key}.one`);
  return t(`${key}.many`, { count });
}
```

Then in JSON:

```json
{ "items": { "zero": "No items", "one": "1 item", "many": "{count} items" } }
```

For complex plural rules (Slavic languages, Arabic), use `Intl.PluralRules` to pick the right key. Skip until you support those locales.

## Never put copy in code

Even one string. The moment you write `<Text>Save</Text>` instead of `<Text>{t("common.save")}</Text>`, you've split the source of truth. Code review for this.

## Anti-patterns

- ❌ Hardcoded strings in components.
- ❌ Adding to one file, not the other (hook blocks, but don't even try).
- ❌ Using `t()` for unique IDs, slugs, or non-user-facing strings.
- ❌ Putting URLs or emails in i18n files (those don't translate).
- ❌ Translating brand names — keep them in English globally.

## Verifying

```bash
# both files have same key set
diff <(jq -S '[paths(scalars)] | flatten' messages/fr.json | jq 'sort') \
     <(jq -S '[paths(scalars)] | flatten' messages/en.json | jq 'sort')

# every t("...") call has a matching key
grep -rn 't("[^"]*")' app components lib --include="*.tsx" | grep -oE 't\("[^"]+"\)' | sort -u
# manually scan for keys not in your json files
```

The hook does the parity check automatically on every Edit/Write.
