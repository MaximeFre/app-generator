# Architecture

How this template is laid out, and why.

## Top-level layout

```
app-generator/
├── app/                    expo-router routes (screens + layouts)
├── components/             UI components (ui = primitives, {domain}/ = feature)
├── lib/                    state, clients, integrations
├── messages/               i18n (fr.json, en.json)
├── drizzle/                generated migrations (commit them)
├── assets/                 icons, splash, images
├── docs/                   this folder
├── .claude/                Claude Code config (agents, skills, rules, hooks)
├── .planning/              generated by /generate-app — design memory (commit it)
├── CLAUDE.md               root project rules for Claude
├── README.md               landing doc
├── app.json                Expo config (name, slug, plugins, bundle id)
├── eas.json                EAS Build profiles (dev/preview/prod)
├── package.json
├── tsconfig.json
├── tailwind.config.js
├── babel.config.js
├── metro.config.js
├── drizzle.config.ts
└── global.css              design tokens (CSS variables)
```

## The 3-layer separation

```
       ┌─────────────────────────────────────┐
       │  app/   (routes only, no logic)     │
       └────────────┬────────────────────────┘
                    │ imports
                    ▼
       ┌─────────────────────────────────────┐
       │  components/  (UI + composition)    │
       └────────────┬────────────────────────┘
                    │ imports
                    ▼
       ┌─────────────────────────────────────┐
       │  lib/   (state + clients + utils)   │
       └─────────────────────────────────────┘
```

Inverse imports are **forbidden**:
- `lib/` must NOT import from `app/` or `components/`.
- `components/ui/` must NOT import from `components/{domain}/` or `app/`.

Why: keeps the dependency graph one-way, avoids circular imports, makes refactors safe. Enforced by code review (no lint rule yet).

## app/ — routes only

`app/` mirrors expo-router's file convention. Each file = a route. Each `_layout.tsx` = a layout.

```
app/
├── _layout.tsx             root: providers (gesture handler, safe area, splash)
├── index.tsx               redirect → /(tabs)
├── (tabs)/
│   ├── _layout.tsx         <Tabs> bottom tabs
│   ├── index.tsx           Home
│   └── settings.tsx        Settings
├── auth/
│   ├── _layout.tsx         <Stack>
│   ├── sign-in.tsx
│   └── sign-up.tsx
├── paywall.tsx             modal route
└── +not-found.tsx          404
```

Rules:
- Routes contain ONLY route logic (params, redirects, mounting components).
- All visible strings → `t("key")`.
- All styling → NativeWind classes.
- Never put business logic, fetch loops, or sync engines inside a route file.

## components/

Two zones:

```
components/
├── ui/                     ← primitives. Universal, no domain.
│   ├── Button.tsx
│   ├── Input.tsx
│   └── Screen.tsx
├── paywall/                ← feature components.
│   └── PremiumGate.tsx
└── forms/                  ← (placeholder for future domain components)
```

**Primitives** (`components/ui/`) — used everywhere. Edit only when the change is universal.
**Domain** (`components/{domain}/`) — feature-specific. Tied to a use case (paywall, capture, history…).

## lib/ — state, clients, integrations

One folder per concern:

```
lib/
├── env.ts                  env reading + featureFlags
├── utils.ts                tiny helpers (cn, etc.)
├── db/
│   ├── client.ts           drizzle + expo-sqlite singleton
│   ├── schema.ts           tables (Drizzle definitions)
│   └── sync.ts             push/pull engine
├── supabase/client.ts      Supabase JS singleton + secure-store adapter
├── store/auth.ts           Zustand: session, user, status
├── sentry/client.ts        init + reportError + setUserContext
├── analytics/
│   ├── events.ts           ALLOWLIST + types (source of truth)
│   └── posthog.ts          init + track + identify + reset
├── revenuecat/
│   ├── client.ts           init + login + restore + refresh
│   └── entitlements.ts     Zustand: isPremium
└── i18n/index.ts           t() function + locale state
```

Each module:
- Singleton via module-level `let cached: X | null = null` + `getX()`.
- Async init = explicit `init{Name}()` called from `app/_layout.tsx`.
- Types exported alongside.

## How they wire up at boot

`app/_layout.tsx` is the boot sequence:

```
initSentry()                 (sync, before everything else)
SplashScreen.preventAutoHide()
↓
useEffect:
  ensureMigrations()         (Drizzle: run pending migrations)
  initAnalytics()            (PostHog: optional, no-op if no key)
  initRevenueCat()           (RC: optional, no-op if no key)
  useAuth.init()             (Supabase: optional, no-op if no key)
  track("app_opened", { from_background: false })
  SplashScreen.hideAsync()
↓
<Stack> with screens
```

Order matters: migrations BEFORE any query, init BEFORE any track call.

## Feature flags

`lib/env.ts` exposes `featureFlags`:

```ts
{
  cloudSyncEnabled: boolean,    // EXPO_PUBLIC_SUPABASE_URL set
  paywallEnabled: boolean,      // EXPO_PUBLIC_RC_*_API_KEY set
  analyticsEnabled: boolean,    // EXPO_PUBLIC_POSTHOG_API_KEY set
  crashReportingEnabled: boolean, // EXPO_PUBLIC_SENTRY_DSN set
}
```

Code paths that touch a third-party service MUST check the flag and no-op gracefully if disabled. The app must run with all four flags `false` (true free tier).

## Free vs premium boundary

The boundary is enforced in TWO places:

1. **UI**: `<PremiumGate feature="x">{premiumOnly}</PremiumGate>` — wraps a section. Renders an upgrade card when not premium.
2. **Data writes**: `lib/db/sync.ts` short-circuits if `!useEntitlements.getState().isPremium`.

Free users can still SEE everything that doesn't require cloud — they just can't sync, export, or get long history. Local data is unlimited.

## Environment / config split

```
.env.example       template for local .env (gitignored)
.env               local dev (gitignored)
EAS env            cloud builds — `eas env:create --visibility public/encrypted`
```

Public-by-design (bundled into JS, visible in IPA/APK):
- `EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_ANON_KEY`
- `EXPO_PUBLIC_RC_IOS_API_KEY`, `EXPO_PUBLIC_RC_ANDROID_API_KEY`
- `EXPO_PUBLIC_POSTHOG_API_KEY`, `EXPO_PUBLIC_POSTHOG_HOST`
- `EXPO_PUBLIC_SENTRY_DSN`

Server-only (NEVER `EXPO_PUBLIC_*`):
- `SUPABASE_SERVICE_ROLE_KEY` (Edge Functions only)
- `SENTRY_AUTH_TOKEN` (build-time, source map upload)

## Native side (when building)

`app.json` plugins generate native code on `npx expo prebuild`:
- `expo-router` → routing
- `expo-secure-store` → Keychain / Keystore (Supabase auth tokens)
- `expo-localization` → device locale
- `expo-sqlite` → native SQLite binding
- `@sentry/react-native/expo` → native Sentry SDK + source map upload

Adding a plugin = `npx expo prebuild --clean` + rebuild dev client. JS-only changes don't need this.
